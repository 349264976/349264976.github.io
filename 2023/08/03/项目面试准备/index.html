  • threadLocal

    ThreadLocal 是 Java 中的一个类，它提供了一种创建线程本地变量的方式。线程本地变量是指只属于特定线程的变量，不与其他线程共享。

    相当于每个线程都有自己的副本 自己的东西自己访问

    private static ThreadLocal threadLocal=new ThreadLocal();
    public static void setCurrentId(Long id) {
    threadLocal.set(id);
    }
    public static Long getCurrentId() {
    return threadLocal.get();
    }

    这段代码定义了一个全局异常处理器 GlobalExceptionHandler，它使用了 Spring 的 @ControllerAdvice 注解来声明，能够处理所有带有 @RestController 或 @Controller 注解的类的异常。

    在 GlobalExceptionHandler 中，定义了两个异常处理方法：

    1. getErrorMessage()：用于处理 SQLIntegrityConstraintViolationException 异常。如果该异常的错误信息中包含 “Duplicate entry”，则说明是由于违反唯一性约束而导致的异常，此时会提取出重复键值，并返回一个包含错误信息的 R 对象。否则，会返回一个包含默认错误信息的 R 对象。
    2. getCustomException()：用于处理自定义异常 CustomException。该方法会直接将异常信息封装成一个包含错误信息的 R 对象，并返回给调用方。

    这些异常处理方法都使用了 @ExceptionHandler 注解来声明，用于指定处理哪种类型的异常。当程序抛出相应的异常时，Spring MVC 会自动将异常对象传递给对应的异常处理方法，并执行该方法中的逻辑。

    需要注意的是，这些异常处理方法都使用了 @ResponseBody 注解来指定返回值类型为 JSON 格式。这意味着当这些方法返回一个 R 对象时，Spring MVC 会自动将该对象序列化成 JSON 格式并返回给调用方。由于这些方法都是返回一个包含错误信息的 R 对象，因此它们都能够提供统一的异常处理机制，从而提高程序的稳定性和可维护性。

    package com.atheima.reggie.common;
import java.sql.SQLIntegrityConstraintViolationException;
import lombok.extern.slf4j.Slf4j;
import org.springframework.stereotype.Controller;
import org.springframework.web.bind.annotation.ControllerAdvice;
import org.springframework.web.bind.annotation.ExceptionHandler;
import org.springframework.web.bind.annotation.ResponseBody;
import org.springframework.web.bind.annotation.RestController;

@ControllerAdvice(annotations = {RestController.class,Controller.class})
@ResponseBody
@Slf4j
public class GlobalExceptionHandler {
    /**
     * 异常处理方法
     * @return
     */
    @ExceptionHandler(SQLIntegrityConstraintViolationException.class)
    public R<String> getErrorMessage(SQLIntegrityConstraintViolationException exception){
        log.error(exception.getMessage());
        if (exception.getMessage().contains("Duplicate entry")){

            String[] split = exception.getMessage().split(" ");
            String error= split[2]+"已经存在";

            return R.error(error);
        }
        return R.error("未知的错误发生");
    }


    /**
     * 异常处理方法
     * @return
     */
    @ExceptionHandler(CustomException.class)
    public R<String> getCustomException(CustomException exception){
        log.error(exception.getMessage());
        return R.error(exception.getMessage());
    }
}

    这段代码是一个 MyBatis Plus 的配置类，用于配置 MyBatis Plus 的插件。

    在这个配置类中，通过 @Configuration 注解将该类标识为一个配置类，告诉 Spring 容器需要对该类进行处理。

    然后，使用 @Bean 注解定义了一个名为 mybatisPlusInterceptor 的方法，该方法返回一个 MybatisPlusInterceptor 实例。

    在方法内部，创建了一个 MybatisPlusInterceptor 对象，并通过 addInnerInterceptor 方法添加了一个 PaginationInnerInterceptor 内部拦截器。PaginationInnerInterceptor 是 MyBatis Plus 提供的分页插件，用于处理分页查询的逻辑。

    最后，将创建的 MybatisPlusInterceptor 对象返回。

    这样，当 Spring 容器初始化时，会自动调用该配置类中的 mybatisPlusInterceptor 方法，将返回的 MybatisPlusInterceptor 实例注册到容器中，使得 MyBatis Plus 插件生效，其中包括了分页查询的功能。

    @Configuration
public class MybatisPlusConfig {
    @Bean
    public MybatisPlusInterceptor mybatisPlusInterceptor(){
        MybatisPlusInterceptor mybatisPlusInterceptor=new MybatisPlusInterceptor();
        mybatisPlusInterceptor.addInnerInterceptor(new PaginationInnerInterceptor());
        return mybatisPlusInterceptor;
    }

}

    这段代码是一个 Redis 的配置类，用于配置 RedisTemplate。

    在这个配置类中，使用 @Configuration 注解将该类标识为一个配置类，告诉 Spring 容器需要对该类进行处理。

    继承了 CachingConfigurerSupport 类，表示扩展了 Spring 的缓存配置支持。

    然后，使用 @Bean 注解定义了一个名为 redisTemplate 的方法，该方法返回一个 RedisTemplate<Object, Object> 实例。

    在方法内部，创建了一个 RedisTemplate<Object, Object> 对象，并进行了一些配置操作。具体配置如下：

    • 通过 setKeySerializer 方法设置了 key 的序列化器为 StringRedisSerializer，将 key 序列化为字符串。
    • 可以根据需要，通过 setValueSerializer 方法设置 value 的序列化器。在代码中，该行代码是被注释掉的，表示使用默认的序列化器 JdkSerializationRedisSerializer。

    最后，将创建的 RedisTemplate<Object, Object> 对象返回。

    这样，当 Spring 容器初始化时，会自动调用该配置类中的 redisTemplate 方法，将返回的 RedisTemplate 实例注册到容器中，使得在应用程序中可以使用该实例来操作 Redis 数据库。

    @Configuration
public class RedisConfig extends CachingConfigurerSupport {

    @Bean
    public RedisTemplate<Object, Object> redisTemplate(RedisConnectionFactory connectionFactory) {
        RedisTemplate<Object, Object> redisTemplate = new RedisTemplate<>();
        //默认的Key序列化器为：JdkSerializationRedisSerializer
        redisTemplate.setKeySerializer(new StringRedisSerializer()); // key序列化
        //redisTemplate.setValueSerializer(new GenericJackson2JsonRedisSerializer()); // value序列化
        redisTemplate.setConnectionFactory(connectionFactory);
        return redisTemplate;

    }
}

controller注解

@RequestBody 注解用于指示 Spring MVC 控制器方法参数绑定时，从请求的主体中获取数据并进行反序列化。

一般情况下，当请求的数据以 JSON、XML 或其他格式作为请求主体发送到服务器时，您需要使用 @RequestBody 注解来告诉 Spring 框架将请求主体中的数据绑定到方法参数上。

在您提供的代码示例中，@RequestBody 注解用于将请求主体中的 JSON 数据反序列化为 Employee 对象，并将其作为参数传递给 login 方法。

使用 @RequestBody 注解的好处是，它使得控制器方法的参数绑定变得简单和自动化。您无需手动从请求中提取数据、解析 JSON 或处理其他反序列化逻辑，Spring 框架会自动完成这些操作。

需要注意的是，使用 @RequestBody 注解时，需要确保请求的 Content-Type 与请求主体中的数据格式相匹配，以便 Spring 框架能够正确解析请求主体中的数据并进行反序列化。常见的 Content-Type 类型包括 application/json、application/xml 等。

分页知识

这段代码是一个处理分页查询的方法，使用了 @GetMapping 注解标记为处理 GET 请求的方法。

请求路径为 “/page”，接收的请求参数包括 page（当前页码）、pageSize（每页显示的记录数）和 name（查询条件）。

方法内部进行了以下处理逻辑：

  1. 使用日志记录了请求中的 page、pageSize 和 name 参数的值。
  2. 构建了一个 Page 对象，表示分页查询的信息，其中传入了当前页码和每页显示的记录数。
  3. 构建了一个 LambdaQueryWrapper 对象，用于构造查询条件。
  4. 使用 like 方法设置了一个查询条件，即根据 name 参数进行模糊查询。当 name 参数不为空时，才会添加该查询条件。
  5. 使用 orderByDesc 方法设置了一个排序条件，即按照 updateTime 字段进行降序排序。
  6. 调用 employeeService 的 page 方法执行分页查询，将查询结果存入 pageinfo 对象中。
  7. 返回一个包装了结果的 R 对象，其中包含了处理结果的状态码和消息，以及分页查询的结果 pageinfo。

最终，返回的结果中包含了分页查询的数据，可以在客户端进行展示或其他操作。

  @GetMapping("/page")
    public R<Page> page(int page,int pageSize,String name) {
        log.info("page={},pageSize={},name={}",page,pageSize,name);
        //构建分页构造器
        Page pageinfo=new Page<>(page,pageSize);
        //构造条件构造器
        LambdaQueryWrapper<Employee> lambdaQueryWrapper=new LambdaQueryWrapper();
        lambdaQueryWrapper.like(StringUtils.isNotEmpty(name),Employee::getName,name);
        lambdaQueryWrapper.orderByDesc(Employee::getUpdateTime);
        //执行查询
        employeeService.page(pageinfo,lambdaQueryWrapper);
        return R.success(pageinfo);
    }
